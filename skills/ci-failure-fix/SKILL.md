# CI Failure Fix

## 目的

CIの失敗ログを読み、原因を分類し、修正方針を提案する。

## 使う場面

- GitHub ActionsなどのCIが失敗した。
- テスト、lint、型チェック、ビルドの失敗原因を調べる。
- ローカルで再現する手順を作る。
- 失敗が環境要因かコード変更によるものか切り分ける。

## 使わない場面

- CIではなく本番バグの調査が主目的の場合。必要なら `bug-investigation` を使う。
- コード差分全体のレビューが主目的の場合。必要なら `code-review` を使う。
- 外部サービスの管理画面操作やAPI連携が必要な場合。

## 確認する入力

- CIログ、失敗したジョブ名、失敗したステップ。
- 対象ブランチ、直近の変更、実行コマンド。
- ローカル環境での再現結果、関連する設定ファイル。

## 手順

1. 失敗したジョブとステップを特定する。
2. ログの最後だけでなく、最初に失敗したエラーを探す。
3. 失敗をテスト、lint、型チェック、ビルド、依存関係、環境、タイムアウトに分類する。
4. 直近の変更と失敗箇所の関連を確認する。
5. ローカルで再現できる最小コマンドを提案する。
6. 修正方針を、コード修正、テスト修正、設定修正、依存関係修正に分けて整理する。
7. flaky testの可能性がある場合は、根拠と安定化の方法を書く。flaky testは、同じ条件でも成功と失敗が揺れるテストを指す。
8. 修正後に実行すべき確認コマンドをまとめる。

## 安全上の注意

- CIを通すためだけにテストを弱めない。仕様上不要なテストかどうかを確認する。
- 破壊的変更や設定削除が必要な場合は、影響範囲と戻し方を書く。
- 依存関係追加やバージョン変更は、理由、影響、ロックファイル更新の有無を明示する。
- 仕様変更でテスト期待値を変える場合は、根拠を明示する。
- テスト不足が判明した場合は、追加すべき確認を提案する。
- ログ内の秘密情報、トークン、認証情報を出力しない。
