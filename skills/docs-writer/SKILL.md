# Docs Writer

## 目的

README、使い方、設計メモ、運用手順などの開発ドキュメントを作成・改善する。

## 使う場面

- READMEを整備する。
- 新機能の使い方を書く。
- 開発者向けドキュメントを追加する。
- 設計判断、運用手順、トラブルシュートを整理する。

## 使わない場面

- コード差分のレビューが主目的の場合。必要なら `code-review` を使う。
- PR本文だけを作りたい場合。必要なら `pr-summary` を使う。
- リリースノートを作りたい場合。必要なら `release-notes` を使う。

## 確認する入力

- 対象読者、目的、既存ドキュメント、関連コード。
- 実際のコマンド、設定値、制約、動作確認結果。
- 公開範囲、書いてはいけない情報、更新すべき関連ページ。

## 手順

1. 読者を確認する。例: 初回利用者、開発者、レビュアー、運用者。
2. 読者が達成したい作業を1つずつ整理する。
3. 既存ドキュメントの構成、用語、表記ゆれを確認する。
4. 手順、前提条件、入力、出力、失敗時の対処を具体的に書く。
5. コードや設定に基づき、実際に存在する内容だけを書く。
6. 仕様変更や未確定事項がある場合は、確定情報と分けて書く。
7. 関連するREADME、docs、コメントの更新漏れを確認する。
8. 読み返して、冗長な説明、曖昧な表現、古い情報を削る。

## 安全上の注意

- 秘密情報、個人情報、内部URL、トークン、パスワードを含めない。
- 破壊的変更を伴う手順には、事前確認、バックアップ、影響範囲を書く。
- 依存関係追加がある場合は、理由とインストール方法を明示する。
- 仕様変更や制限事項を隠さない。
- テスト不足や未検証の手順は、確認済みのように書かない。

## 出力形式

```markdown
## 対象読者

## 目的

## 前提条件

## 手順

1. 

## 確認方法

## 注意点

## 関連情報

## 未確認事項
```
