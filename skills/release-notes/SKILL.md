# Release Notes

## 目的

変更内容からリリースノートを作成する。

## 使う場面

- リリース前に変更点を整理する。
- ユーザー向けと開発者向けの変更点を分けて書く。
- breaking changesを明示する。breaking changesは、既存利用者の対応が必要な互換性のない変更を指す。
- バグ修正、改善、ドキュメント変更を分類する。

## 使わない場面

- PR単体の説明文を作りたい場合。必要なら `pr-summary` を使う。
- READMEや利用手順などの開発ドキュメントを書きたい場合。必要なら `docs-writer` を使う。
- 変更差分がなく、リリース対象が分からない場合。
- マーケティング文章を作ることが主目的の場合。

## 確認する入力

- リリース対象の差分、コミット、PR一覧、変更ログ。
- 対象バージョン、リリース日、対象ユーザー。
- 既知の制限、移行手順、互換性への影響。

## 手順

1. リリース対象の変更を確認し、重複や内部的すぎる変更を整理する。
2. ユーザーに影響する変更と、開発者や運用者に影響する変更を分ける。
3. 追加、変更、修正、削除、ドキュメント、内部改善に分類する。
4. breaking changesがある場合は、影響、必要な対応、移行手順を明示する。
5. 破壊的変更やデータ移行がある場合は、事前確認とロールバック注意点を書く。
6. 未確認の変更やテスト不足がある場合は、既知の注意点として分ける。
7. 読み手に合わせて、簡潔で具体的な表現に整える。

## 安全上の注意

- 未リリース機能、秘密情報、内部事情を不用意に含めない。
- 破壊的変更は目立つ場所に書き、影響を曖昧にしない。
- 依存関係追加や削除が利用者に影響する場合は明示する。
- 仕様変更を単なる修正として書かない。
- テスト不足や既知の制限がある場合は、隠さず分けて書く。

## 出力形式

```markdown
# Release Notes

## Highlights

- 

## Added

- 

## Changed

- 

## Fixed

- 

## Breaking Changes

- 影響:
- 必要な対応:

## Known Issues

- 
```
