# Skill Reviewer

## 目的

このリポジトリ内の `SKILL.md` をレビューし、構成、description、安全性、使いやすさを確認する。

## 使う場面

- 新しいSkillを追加した。
- 既存Skillを改善する。
- descriptionが曖昧でないか確認する。
- 手順や安全上の注意が十分か確認する。

## 使わない場面

- 一般的なコードレビューを行う場合。必要なら `code-review` を使う。
- 通常のMarkdown文書やREADMEの文章改善が主目的の場合。必要なら `docs-writer` を使う。
- 外部サービス連携Skillを作る場合。この初期構成では対象外とする。

## 確認する入力

- 対象の `SKILL.md`。
- Skillの目的、想定する依頼、対象外の作業。
- 参考元がある場合は、出典とライセンス情報。
- READMEのSkill一覧との整合性。

## 手順

1. YAML frontmatterがあり、`name` と `description` が含まれているか確認する。
2. `name` が小文字英数字とハイフンのみのkebab-caseか確認する。
3. `description` に、いつ使うべきか分かる具体的なトリガー語が含まれているか確認する。
4. 本文に「目的」「使う場面」「使わない場面」「確認する入力」「手順」「安全上の注意」「出力形式」があるか確認する。
5. 手順が番号付きリストで、Codexがそのまま作業できる粒度か確認する。
6. 破壊的変更、依存関係追加、仕様変更、テスト不足、秘密情報の扱いへの注意があるか確認する。
7. 1つのSkillが複数の作業を抱え込みすぎていないか確認する。
8. 参考元がある場合は、ライセンスと出典が `NOTICE.md` または該当Skill内に記録されているか確認する。
9. READMEのSkill一覧更新が必要か確認する。

## 安全上の注意

- 公開Skillの本文をコピーしていないか確認する。
- ライセンス不明の内容を取り込まない。
- scriptsを追加する場合は、目的、危険性、実行条件が明記されているか確認する。
- 初期構成ではscriptsを作らない方針に反していないか確認する。
- 外部サービス連携、API連携、MCP連携、SaaS連携が紛れ込んでいないか確認する。
- 秘密情報や実在の認証情報が例示されていないか確認する。

## 出力形式

```markdown
## レビュー結果

- 判定: OK / 要修正

## 指摘

- [重大度] 対象箇所:
  問題:
  提案:

## 良い点

- 

## README更新の要否

- 

## ライセンス・出典確認

- 

## 残るリスク

-
```
