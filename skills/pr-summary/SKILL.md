# PR Summary

## 目的

変更差分からPull Request本文を作成する。

## 使う場面

- PR説明文を書く。
- 変更点、背景、確認方法、影響範囲を整理する。
- レビュアー向けに要点をまとめる。
- 実装後にPRテンプレートへ入れる内容を作る。

## 使わない場面

- コードの問題点を詳しくレビューしたい場合。必要なら `code-review` を使う。
- リリースノートを作りたい場合。必要なら `release-notes` を使う。
- 変更内容が確認できない場合。

## 確認する入力

- 差分、コミット、変更ファイル、関連仕様。
- 変更の背景、目的、ユーザーへの影響。
- 確認済みテスト、未確認項目、スクリーンショットの有無。

## 手順

1. 差分を確認し、実際に変更された内容だけを抽出する。
2. 背景と目的を、推測ではなく確認できる情報からまとめる。
3. レビュアーが最初に見るべき重要な変更点を箇条書きにする。
4. 影響範囲を、画面、API、DB、設定、テスト、ドキュメントなどに分けて確認する。
5. 破壊的変更や仕様変更があれば明示する。
6. 実行済みの確認方法と、未実行の確認方法を分けて書く。
7. レビュアーへの依頼事項があれば短く書く。

## 安全上の注意

- 実装していない内容や未確認の効果をPR本文に書かない。
- 破壊的変更がある場合は、移行手順やロールバック注意点を明示する。
- 依存関係追加がある場合は、理由と影響を説明する。
- 仕様変更は「修正」や「改善」と混ぜずに明示する。
- テスト不足や未確認事項を隠さない。
- 秘密情報、内部URL、トークン、個人情報をPR本文に含めない。

## 出力形式

```markdown
## 概要

## 変更点

- 

## 背景

## 影響範囲

- 

## 確認方法

- [x] 
- [ ] 

## レビュアーへの補足

- 
```
