# Code Review

## 目的

コード差分やPull Requestをレビューし、バグ、設計リスク、保守性、セキュリティ、テスト不足を確認する。

## 使う場面

- PRレビューを行う。
- `git diff` や変更ファイルを確認する。
- 実装後にセルフレビューを行う。
- 変更前にリスクを洗い出す。

## 使わない場面

- まだレビュー対象のコードや差分がない場合。
- PR本文だけを作りたい場合。必要なら `pr-summary` を使う。
- テストの設計だけを詳しく行いたい場合。必要なら `test-writer` を使う。

## 確認する入力

- レビュー対象の差分、変更ファイル、PR説明、関連Issueや仕様。
- 変更の目的、想定ユーザー、影響範囲。
- 実行済みテスト、未実行テスト、既知の制約。

## 手順

1. 変更目的と差分全体を確認し、触っている機能と境界を把握する。
2. バグにつながる条件分岐、例外処理、状態更新、非同期処理、データ変換を優先して読む。
3. セキュリティ上の問題がないか確認する。特に入力検証、認可、秘密情報、ログ出力、外部入力の扱いを見る。
4. 既存設計との一貫性、責務分離、重複、命名、将来の保守性を確認する。
5. 仕様変更が含まれていないか確認し、含まれる場合は明示する。
6. テストが変更内容とリスクを十分に覆っているか確認する。
7. 重大度の高い指摘から順に、ファイルと行番号を添えて簡潔にまとめる。
8. 確認できなかった点、追加で必要なテスト、残るリスクを最後に書く。

## 安全上の注意

- 破壊的変更が含まれる場合は、データ消失や後方互換性への影響を必ず確認する。
- 依存関係追加がある場合は、必要性、ライセンス、保守状況、サイズ、セキュリティリスクを確認する。
- 仕様変更がレビュー中に見つかった場合は、実装ミスと仕様判断を分けて書く。
- テスト不足は、どのリスクが未検証なのか具体的に書く。
- APIキー、トークン、パスワードなどの秘密情報がコード、設定、ログ、ドキュメントに含まれていないか確認する。

## 出力形式

重大度順に指摘を書く。

```markdown
## 指摘

- [重大度] `path/to/file.ext:line`
  問題:
  影響:
  提案:

## 確認事項

- 確認したい点を書く。

## テスト観点

- 追加または確認すべきテストを書く。
```
